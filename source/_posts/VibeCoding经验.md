## 前言

VibeCoding（AI 辅助编程）正在改变我们的开发方式。它不是让 AI 完全替代程序员，而是让人和 AI 高效协作，快速将想法变为现实。

本文分享我在实践中的经验，从基础概念到高级技巧，帮助你更好地驾驭 AI 编程。

---

## 一、需求分析：先想清楚再动手

在 AI 编程中，需求分析同样重要。抓住 **用户痛点** 和 **使用场景**，才能让 AI 产出真正有价值的代码。

| 维度 | 问题 |
|:----:|:-----|
| 目标用户 | 谁会用这个工具？ |
| 核心价值 | 解决什么问题？ |
| 产品形态 | 用什么方式呈现？ |
| 技术可行性 | 能否实现？ |
| 优先级 | 先做什么？ |

---

## 二、敏捷开发：小步快跑

AI 编程天然适合敏捷开发模式，尤其适合以下场景：

- **需求明确且轻量**：不是复杂的企业级系统，而是工具类产品
- **快速迭代**：先做出 MVP（最小可行产品），再逐步优化
- **用户即开发者**：自己就是用户，能快速验证和反馈

核心理念：**先做出能用的版本，再通过多轮迭代不断完善**。

## 三、核心工具：MCP 与 Skills

### MCP：AI 的手和眼睛

MCP（Model Context Protocol，模型上下文协议）是一套标准接口，让 AI 能够与外部工具和数据源交互。

| 能力 | 示例 |
|:----:|:-----|
| 连接外部工具 | 搜索网页 |
| 获取实时数据 | 读取代码 |
| 操作外部系统 | 查询数据库 |

**常用 MCP：**
- **Firecrawl MCP**：抓取和解析网页内容
- **Context7 MCP**：获取最新技术文档，避免使用过时的 API

### Agent Skills：AI 的技能包

Skills 是给 AI 准备的技能包，让 AI 快速学习使用各种专业技能，无需每次重复输入提示词。

### MCP vs Skills 对比

| 维度 | MCP | Agent Skills |
|:----:|:----|:-------------|
| 本质 | 通信协议，连接 AI 与外部工具 | 知识包，教 AI 特定技能 |
| 运行方式 | 需要启动服务器进程 | 纯文件，无需额外进程 |
| 内容 | API 接口定义 | 提示词 + 脚本 + 参考资料 |
| 举例 | Firecrawl（网页抓取能力） | UI UX Pro Max（前端美化技能） |

**协作关系：** MCP 提供食材，Skills 提供配方。

### Skills 安装方法

```bash
npx skills add https://github.com/vercel-labs/skills --skill find-skills
```

安装后可让 AI 帮你安装其他技能：

```
请你帮我安装技能 UI UX Pro Max 和 Skill Creator
```

## 四、提示词设计：与 AI 高效沟通

### 基础提示词模板

```
你是一位专业的程序员，现在请你根据需求，帮我设计方案、人工确认、分步骤完成开发、执行测试、找我验收。 
## 需求 
@需求 
## 注意事项 
1）前端页面要足够独特，不要千篇一律 
2）你需要先完成网页版、确保功能正常使用之后，再开发 Agent Skills 
3）项目需要利用 AI，请你利用 xxx 来对接 AI 服务
```

### 关键要素

| 要素 | 说明 |
|:----:|:-----|
| 角色设定 | "你是一位专业的程序员" — 明确 AI 角色定位 |
| 人工确认 | 防止 AI 自作主张，跑偏方向 |
| 开发顺序 | 明确优先级，避免 AI 同时开始多个任务 |
| 技术约束 | 指定具体技术方案，减少 AI 决策负担 |

### 开放式补充技巧

```
你还可以有自己的想法
```

先明确核心需求，再给 AI 一个"自由发挥"的空间，但要求人工确认。AI 可能会提出你没想到但很有价值的展示维度。

## 五、文档沉淀：AI 的持久记忆

AI 的上下文窗口有限，文档可以作为持久化的"记忆"。

### 文档沉淀的好处

1. **解决 AI "断片"问题**：新对话中只需提供文档即可让 AI 快速恢复上下文
2. **提升协作效率**：团队成员可通过文档快速了解项目背景
3. **便于版本追溯**：记录方案演进过程
4. **降低沟通成本**：Markdown 格式清晰易读

### 建议沉淀的文档

- 需求文档：明确要做什么
- 方案设计文档：明确怎么做
- 技术选型说明：记录为什么选择某项技术

### 沉淀提示词

```
请你把需求和方案都沉淀为 Markdown 文档，然后再进行开发，注意，你必须利用 MCP 获取到最新的 API 对接和技术实现方法，防止使用过时的代码
```

## 六、AI 编码工作流

AI 收到"开始开发"指令后，会按以下流程执行：

```
规划任务列表 → 创建项目结构 → 安装依赖 → 编写后端代码 → 编写前端代码 → 配置数据库 → 启动项目
```

**自动完成的工作：**
1. 项目初始化：创建 `package.json`、配置 TypeScript、设置 ESLint
2. 后端开发：搭建服务器、配置路由、实现业务逻辑
3. 前端开发：创建前端项目、编写组件、配置路由
4. 数据库：通过 ORM 定义数据模型、自动生成迁移

## 七、版本控制：AI 编程的安全网

在 AI 编程中，版本控制尤为重要，因为 AI 的改动可能是大范围的。

### 最佳实践

1. **每完成一个功能就提交一次**：方便快速回滚
2. **提交信息要清晰**：如 `feat: 完成基础功能开发`
3. **改动前先提交当前状态**：确保当前可用代码不会丢失
4. **善用分支**：对于大的改动（如 UI 重构），在新分支上进行

## 八、前端优化：消除 AI 味

AI 生成的网页常有"AI 味"：蓝紫色渐变背景、圆角卡片、通用图标。这是因为大模型在训练时看到了大量类似风格的设计。

### 消除方法

1. **指定色彩方案**：明确告诉 AI 用什么颜色
2. **使用特定 UI 库**：如 Aceternity UI，有独特的视觉风格
3. **注入情感和场景**：告诉 AI 产品的调性（科技感、极客风等）
4. **使用前端美化技能**：如 UI UX Pro Max

### AI 工具协作示例

- **UI UX Pro Max 技能**：提供专业的 UI/UX 设计指南
- **Context7 MCP**：获取 Aceternity UI 最新的组件文档
- **代码编辑能力**：根据设计方案修改组件代码

## 九、AI 中断处理

### 中断原因

1. 网络不稳定
2. 上下文过长
3. 任务过大

### 应对策略

| 方案 | 说明 | 适用场景 |
|:----:|:-----|:---------|
| 拆分任务 | 把大任务拆成多个小任务 | 改动范围大 |
| 换模型 | 切换到更稳定的大模型 | 特定模型频繁断连 |
| 新开对话 | 重新开一个对话窗口 | 上下文过长 |
| 提供记忆 | 通过提示词帮 AI 找回记忆 | 新对话中继续之前的任务 |

### 恢复记忆提示词

```
你是一位专业的程序员，现在请你根据我的需求和方案、以及已经开发完成的前端和后端项目，在此基础上进行优化和功能扩展。

## 提供给你的资料
@需求文档
@方案设计文档
必须完整分析前端和后端已有代码
```

## 十、高级技巧

### 避免套娃优化

"套娃优化"是 AI 编程中常见的陷阱：修复问题 A → 引入问题 B → 修复问题 B → 问题 A 又回来了。

**解决策略：**
- 设定明确的优先级（如质量 > 数量）
- 让 AI 在修改前先完整理解当前代码逻辑
- 如果 3 次修复还不能解决，考虑新开上下文或回滚代码

### 让 AI 设计功能维度

人工设计筛选排序维度时，容易遗漏或过于主观。AI 可以根据数据模型中已有的字段，自动分析出所有可能的筛选和排序维度。

### AI 编写单元测试

让 AI 编写单元测试来验证代码：
1. 自动验证：不需要人工逐个测试
2. 防止回归：后续修改时自动检测是否破坏逻辑
3. 文档作用：测试代码本身就是逻辑的最好说明

### UI Bug 排查技巧

1. 截图给 AI
2. 描述具体现象（如"筛选栏被帖子内容遮挡了"）
3. 提到 CSS 关键词（如 `z-index`、`overflow`、`position`）
4. 使用浏览器开发者工具（F12）检查元素样式

### 人工确认方案

让 AI 出方案，人工确认后再写代码，比 AI 直接开写然后你再改，效率高得多。

**核心检查点：**
1. 技术栈是否合理？有没有过度引入不必要的依赖？
2. 方案是否足够简单？能不能用更少的代码实现同样的效果？
3. AI 是否理解了你的真实需求？
4. 开发步骤是否清晰？能不能分步骤验证？

### 利用开源项目

在做技术选型时，"站在巨人的肩膀上"非常重要。

**寻找和评估流程：**
GitHub 搜索关键词 → 按 Star 数排序 → 查看 README 和文档 → 看最近更新频率 → 跑一下 Demo 验证 → 确认开源协议

**重点关注：**
- Star 数量（社区认可度）
- 最近一次更新时间（是否还在维护）
- Issue 处理速度（社区活跃度）
- 开源协议（是否允许商用）

## 结语

AI 编程的核心不是让 AI 替代你思考，而是让 AI 帮你快速实现想法。掌握正确的协作方式，你就能成为更高效的开发者。

**记住：你不需要懂设计，只需要会表达需求。通过精准的提示词 + 合适的技能包，AI 就能产出专业级的成果。**
