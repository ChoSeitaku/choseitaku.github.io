## 一、Redis 的五大数据类型以及底层结构

### 

| 数据类型 | 底层结构 | 典型场景 |
|----------|---------|----------|
| **String** | 动态字符串（SDS） | 缓存、计数器、分布式锁 |
| **Hash** | 哈希表（小数据量用压缩列表） | 存储对象（如用户信息） |
| **List** | 双向链表（小数据量用压缩列表） | 消息队列、时间线 |
| **Set** | 哈希表 / 整数集合（IntSet） | 去重、共同好友、标签 |
| **Sorted Set** | 跳表 + 哈希表 | 排行榜、优先级队列 |

**面试关键：** 每种类型都说出一个典型使用场景 + 底层结构。

## 二、Redis 为什么这么快

### 

**六大原因，按重要性排序：**

1. **内存存储**：数据在内存中，比磁盘快几个数量级
2. **高效网络模型（核心）**：单线程 + IO 多路复用（epoll），自定义 RESP 协议简洁高效
3. **单线程模型**：避免线程切换和锁竞争开销
4. **高效数据结构**：跳表、压缩列表等经过大量优化
5. **异步持久化**：fork 子进程做 RDB/AOF，不阻塞主线程
6. **优化的算法**：如近似 LRU 淘汰、渐进式 rehash

> **一句话：** Redis 的快 = 内存 + 单线程 + IO 多路复用 + 高效数据结构。

## 三、Redis 的持久化方案和各自优缺点

### 

**RDB（快照）：** 定时将内存数据快照保存为 `.rdb` 二进制文件

| 优点 | 缺点 |
|------|------|
| 文件小，恢复快 | 可能丢失最后一次快照之后的数据 |
| 适合冷备份 | fork 子进程时性能开销大 |

**AOF（追加日志）：** 记录每条写命令到日志文件，重启时重放

| 优点 | 缺点 |
|------|------|
| 数据安全性高（最多丢 1 秒数据） | 文件体积大 |
| 日志可读，方便排查 | 恢复速度慢 |
| 有重写机制压缩日志 | 性能开销较大 |

**选型建议：**
- 允许少量数据丢失 → 只用 RDB
- 数据安全要求高 → 开启 AOF，`appendfsync everysec`
- 最佳实践 → RDB + AOF 混合持久化（Redis 4.0+）

## 四、Redis 主从模式和集群模式

### 

**主从模式：**
- 主节点负责写，从节点负责读（读写分离）
- 从节点通过 `slaveof` 命令同步主节点数据
- 主从复制流程：全量复制（首次）→ 增量复制（后续）
- **缺点：** 主节点故障需人工切换，不能自动故障转移（需配合哨兵）

**Redis Cluster（集群模式）：**
- 数据分片：16384 个槽（slot），均匀分配到多个主节点
- 每个主节点可配从节点，实现高可用
- 自动故障转移：主节点挂了，从节点自动提升
- 去中心化：任意节点都可接收请求，自动路由到正确节点
- **最少 6 个节点**（3 主 + 3 从）

**面试一句话：** 主从解决读写分离，Cluster 解决数据分片 + 自动故障转移 + 水平扩展。

## 五、Redis 的过期策略

### 

Redis 使用 **惰性删除 + 定期删除** 的混合策略：

| 策略 | 方式 | 优点 | 缺点 |
|------|------|------|------|
| **惰性删除** | 访问 key 时检查是否过期，过期则删 | CPU 友好 | 内存浪费（不访问的过期 key 一直占着） |
| **定期删除** | 每隔 100ms 随机抽取一批 key 检查删除 | 及时释放内存 | CPU 开销（需控制扫描频率和时长） |

**为什么不用定时删除？** 为每个 key 建定时器，大量过期 key 时 CPU 消耗太大。

## 六、Redis 的内存淘汰策略

### 

当内存达到 `maxmemory` 限制时，Redis 按配置的策略淘汰 key：

| 策略 | 淘汰范围 | 淘汰依据 |
|------|---------|----------|
| **noeviction**（默认） | — | 不淘汰，新写入报错 |
| **allkeys-lru** | 所有 key | 最近最少使用 |
| **allkeys-lfu** | 所有 key | 最不经常使用 |
| **volatile-lru** | 设了过期时间的 key | 最近最少使用 |
| **volatile-lfu** | 设了过期时间的 key | 最不经常使用 |
| **allkeys-random** | 所有 key | 随机 |
| **volatile-random** | 设了过期时间的 key | 随机 |
| **volatile-ttl** | 设了过期时间的 key | 越早过期越先淘汰 |

**面试要点：** 缓存场景推荐 `allkeys-lru`；有热点数据场景推荐 `allkeys-lfu`。

## 八、什么是 Nginx，主要有什么作用

### 

**Nginx 是一个高性能的 HTTP 服务器和反向代理服务器**，事件驱动的异步架构使其能处理高并发连接。

**六大作用：**

| 角色 | 说明 |
|------|------|
| **Web 服务器** | 处理静态文件（HTML、CSS、JS、图片） |
| **反向代理** | 隐藏后端服务器，转发请求到应用服务器 |
| **负载均衡** | 将请求分发到多个后端服务器（轮询、IP 哈希、权重） |
| **HTTP 缓存** | 缓存静态内容，减轻后端压力 |
| **SSL/TLS 终止** | 处理 HTTPS 加密解密 |
| **API 网关** | 微服务架构中的统一入口 |

> **面试金句：** Nginx 用事件驱动 + 异步非阻塞解决了 C10K 问题，单机可支撑数万并发连接。

## 九、正向代理和反向代理的区别

### 

**一句话：正向代理隐藏客户端，反向代理隐藏服务器。**

| 维度 | 正向代理 | 反向代理 |
|------|---------|----------|
| 代理对象 | 客户端 | 服务端 |
| 谁被隐藏 | 客户端真实 IP | 后端服务器的真实 IP |
| 典型场景 | VPN、翻墙、企业上网管控 | 负载均衡、API 网关、CDN |
| 客户端感知 | 知道自己用了代理 | 不知道自己访问的是代理 |
| 示例 | Shadowsocks、浏览器代理设置 | Nginx、Cloudflare |

## 十、Nginx 配置文件的结构

### 

按层级记忆：**全局 → events → http → server → location**

```
worker_processes 1;           # 全局：工作进程数

events {
    worker_connections 1024;  # 每个进程最大连接数
}

http {
    server {
        listen 80;            # 监听端口
        server_name xxx.com;  # 域名
        
        location / {
            root /var/www;    # 根目录
            index index.html; # 默认首页
        }
        
        location /api/ {
            proxy_pass http://backend;  # 反向代理
        }
    }
}
```

**关键块说明：**
- `events`：连接数配置
- `http`：HTTP 相关全局配置（gzip、日志格式等）
- `server`：虚拟主机配置（一个站点一个 server 块）
- `location`：URL 匹配规则（决定请求如何处理）

## 十一、Nginx 中配置虚拟主机的要点

### 

配置一个虚拟主机的关键四步：

1. **创建配置文件**：在 `conf.d/` 下创建 `.conf` 文件，或在 `nginx.conf` 的 `http` 块中添加 `server` 块
2. **设置监听和域名**：`listen 80;` + `server_name example.com;`
3. **指定根目录**：`root /var/www/example.com;` + `index index.html;`
4. **配置 location 规则**：处理不同路径（静态资源、反向代理、错误页面）

## 十二、MyBatis 和 MyBatis-Plus 的区别

### 

**一句话：MyBatis-Plus 是 MyBatis 的增强工具，只做增强不做改变。**

| 对比 | MyBatis | MyBatis-Plus |
|------|---------|-------------|
| CRUD 操作 | 需手写 SQL | 内置通用 Mapper/Service，单表 CRUD 零 SQL |
| 条件构造 | 手写动态 SQL | `QueryWrapper` / `LambdaQueryWrapper` 链式调用 |
| 分页 | 需手写分页逻辑 | 内置分页插件，一行代码分页 |
| 代码生成 | 无 | 内置代码生成器 |
| 主键策略 | 手动处理 | 支持多种 ID 策略（雪花算法等） |
| 侵入性 | — | 无侵入，不可影响原有 MyBatis 功能 |

**面试重点：** 提到 Lambda 条件构造、通用 CRUD、代码生成器三个核心卖点。

## 十三、项目用户认证的两种方案及 JWT

### 

**Session 认证（传统方案）：**

```
客户端登录 → 服务器创建 Session 存用户信息 → 返回 SessionID（存 Cookie）
后续请求 → 浏览器自动带 Cookie → 服务器根据 SessionID 查 Session
```

- 缺点：Session 存在服务器内存，分布式需要共享存储；扩展性差

**Token 认证（主流方案）：**

```
客户端登录 → 服务器生成 Token（JWT）签名后返回 → 客户端存储 Token
后续请求 → 客户端请求头带 Token → 服务器验签，无需查存储
```

- 优点：无状态，天然支持分布式；不占用服务器内存

**JWT 结构（三段 Base64，用 `.` 分隔）：**
- **Header**：声明类型（JWT）+ 签名算法（HS256/RS256）
- **Payload**：存放用户信息、过期时间等声明
- **Signature**：对 Header + Payload 用密钥签名，防篡改

> **面试金句：** JWT 的核心优势是无状态，服务器不存 session，天然支持分布式和集群部署。

## 十五、Redis 除了做缓存还能干什么

### 

Redis 的应用场景远超缓存，面试时说出 5 个以上：

| 场景 | 实现方式 |
|------|---------|
| **分布式锁** | `SETNX` + 过期时间（或用 Redisson） |
| **计数器/限流** | `INCR` 实现访问计数、接口限流 |
| **消息队列** | List 的 `LPUSH`/`BRPOP` 或 Stream（5.0+） |
| **排行榜** | Sorted Set 的 `ZADD`/`ZRANGE` |
| **会话共享** | 分布式 Session 集中存储 |
| **发布订阅** | Pub/Sub 实现实时消息推送 |
| **位图统计** | Bitmap 实现签到、在线统计等 |
